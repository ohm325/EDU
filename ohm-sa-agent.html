รอยรั่วของการพัฒนาซอฟต์แวร์ (The Missing Link)

"ปัญหาไม่ได้อยู่ที่โปรแกรมเมอร์เขียนโค้ดไม่เก่ง และไม่ได้อยู่ที่ผู้ใช้งานบอกความต้องการไม่ชัดเจน แต่มันอยู่ที่เราขาด 'คนแปลงสารและวางโครงสร้าง' (System Analyst) ก่อนเริ่มเขียนโปรแกรม"

มุมมองฝั่งธุรกิจ (Business Pain)

"ทำไมได้ของไม่ตรงกับที่สั่ง?"

  • ต้นทุนบานปลาย: โปรแกรมเมอร์เข้าใจผิด เขียนระบบเสร็จแล้วต้องมารื้อทำใหม่ (Rework)
  • ผลกระทบแอบแฝง: ฟีเจอร์ใหม่ไปทำให้ระบบเก่าพัง เพราะไม่มีคนประเมินผลกระทบก่อน
  • ผูกขาดกับคนทำ: พนักงานลาออกทีไร คนใหม่มาทำต่อไม่ได้ เพราะไม่มีเอกสารระบบทิ้งไว้ให้เลย
มุมมองฝั่งไอที (Tech Pain)

"ทำไม Requirement ถึงกำกวม?"

  • AI โปรแกรมเมอร์มโน: ถ้าโยนแค่ "คำสั่งภาษาคน" ให้ AI เขียนโค้ดเลย AI จะคิดชื่อตารางเองมั่วๆ หน้าบ้านกับหลังบ้านจะคุยกันไม่ได้
  • ขาด Test Case ชัดเจน: โปรแกรมเมอร์เขียนโค้ดเสร็จ แต่ไม่รู้ว่าต้องเทสต์ยังไงถึงจะครอบคลุม Logic ที่ลูกค้าขอ

การทำงานร่วมกัน: มนุษย์คุมเกม AI ออกแรง

ในฐานะ SA เราไม่ให้ AI แตะ Database จริง แต่ใช้มันเป็นเครื่องมือร่างโครงสร้างและสรุปบริบท (Contextual Generator)

1. กติกาบนสายพานเดียวกัน (Ecosystem Protocol)

รับ JSON จาก Node 01 (BA) โดยตรง ไม่มีการส่งต่อด้วยไฟล์ Word ป้องกันความผิดพลาดของมนุษย์

2. ประเมินทิศทางโดยมนุษย์ (Human Decision)

[SA Human] ตรวจสอบ Requirement เทียบกับระบบเดิม เพื่อฟันธงว่าเป็น New Module หรือ Customize Module

3. ร่างโครงสร้างเชิงบริบท (Contextual DB Schema)

[SA Agent] ร่าง Entity-Relationship ออกมาเป็นเอกสาร (ไม่ใช่คำสั่งรันจริง) เพื่อให้ SA Human ตรวจสอบกฎเกณฑ์และผลกระทบ (Impact) ก่อน

4. สรุปผลกระทบให้ BA (The Bridge Document)

[SA Agent] สร้างเอกสาร "System Impact Summary" (ภาษาธุรกิจ) ส่งกลับให้ BA เพื่อนำไปอธิบายและขอ Sign-off จากผู้ใช้งาน

กฎเหล็กมาตรฐานระดับองค์กร (Governance)

ทำไม SA Automate ของเราถึงฉลาดและไว้ใจได้? เพราะมนุษย์ (SA Human) เป็นผู้ถือสิทธิ์ขาด ในขณะที่ AI ปฏิบัติตามกฎ 100%

AI เสนอ มนุษย์อนุมัติ

RULE: REVIEW BEFORE EXECUTE

AI ไม่มีสิทธิ์รันคำสั่ง ALTER TABLE ลงฐานข้อมูลโดยตรง SA Human ต้องเป็นผู้อ่านและกดรันด้วยตนเองเสมอ

อิงบริบทระบบเดิม

RULE: CONTEXT INJECTION

AI จะไม่สร้างโมดูลหรือตารางใหม่แบบมโน แต่จะต้องอ้างอิงจากรายชื่อตารางปัจจุบัน (Schema Context) ที่มนุษย์ป้อนให้เสมอ

คุณภาพนำหน้าเสมอ

RULE: TEST-DRIVEN DESIGN

บังคับให้ AI สร้าง Test Case ครอบคลุม Business Logic (Happy & Negative Path) ก่อนที่จะปล่อยสเปกไปให้ทีม Dev เขียนโค้ด

Prompt Factory

SA Agent จะนำ JSON Blueprint + Schema ที่มนุษย์ตรวจแล้ว มาแพ็กกลายเป็น "5 Prompts ระดับองค์กร" เพื่อเป็นพิมพ์เขียวให้แต่ละ Node ในสายพานรับไปทำงานต่อทันที

ผลผลิต 5 รายการจากระบบ SA

1. Spec Summary (For BA & Users)

เอกสารสรุปผลกระทบและข้อจำกัดทางเทคนิค (ภาษาธุรกิจ)

2. DB Schema Prompt (For DBA Agent)

SQL DDL, Data Types, Primary/Foreign Keys, Indexing

3. Backend Prompt (For API Agent)

API Endpoints, Request/Response DTOs, Business Logic Validation

4. Frontend Prompt (For UI Agent)

Component Tree, State Management, API Integration Spec

5. Test Case Prompt (For QA Agent)

Gherkin Syntax (Given-When-Then) ดักทุก Edge Cases
